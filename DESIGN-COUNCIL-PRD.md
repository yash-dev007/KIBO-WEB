# Design Council PRD - KIBO
> Generated by Design Council v1.0 | Date: 2026-05-12
> Panel: Don Norman, Jony Ive, Steve Jobs

## 1. Executive Summary
KIBO's website must describe the real project: a frameless animated desktop companion with voice, neural TTS, persistent memory, privacy controls, and graceful Groq/Ollama/Piper fallbacks. The previous site leaned on editorial mystery and portfolio-like language, which made the product feel less concrete than the markdown describes.

The redesign should lead with the companion itself, then prove the system: voice loop, local memory, proactivity policy, onboarding, diagnostics, performance, and open-source install path.

## 2. Project Analysis
The target audience is developers and AI-tool enthusiasts evaluating a desktop companion that is more personal than a chatbot but more controllable than an always-on assistant. Their core questions are: what is it, what does it access, does it run locally, how fast is it, and can I trust it?

## 3. Design Principles
- Product truth before atmosphere.
- Make the desktop companion visible in the first viewport.
- Treat privacy and memory controls as first-class product features.
- Use proof points from the markdown instead of invented marketing claims.
- Keep the interface warm, but make the hierarchy fast to scan.
- Avoid portfolio navigation for a product website.

## 4. Information Architecture
### 4.1 Navigation structure
Primary navigation should be Features, Architecture, Privacy, and Install.

### 4.2 Content hierarchy
Hero: KIBO lives on your desktop.  
Proof: latency, tests, CPU, memory cap.  
Features: voice, memory, animation, proactivity.  
Architecture: fallback stack.  
Privacy: onboarding, consent, diagnostics, local data.  
Install: GitHub and quick-start commands.

### 4.3 Site map / screen flow
Home anchors should cover the complete product story. Secondary pages can return later only when they map to real docs or release content.

## 5. Visual Design
### 5.1 Recommended visual style
Use a desktop-instrument feel: crisp borders, transparent panels, restrained warmth, and visible companion imagery. Avoid gallery/card-showcase staging.

### 5.2 Color palette
Ink, paper, ash, coral, teal, and white. Teal signals trust/health; coral marks voice/activity; ink grounds the engineering story.

### 5.3 Typography
Keep Inter Tight for product headlines, Inter for body, and JetBrains Mono for runtime/proof details.

### 5.4 Spacing and grid
Use a 1360px max layout, split hero grid, dense proof grid, and full-width bands for major sections.

## 6. Key Components
### 6.1 Main components identified
Hero runtime mock, proof metrics, capability tiles, fallback matrix, trust checklist, install terminal.

### 6.2 Design system recommendations
Prefer square or near-square controls, 8px-or-less card radii, crisp borders, lucide icons for commands, and stable dimensions for proof grids.

### 6.3 Critical states and variants
Show provider health, talking/listening state, memory enabled state, and proactivity off/snoozed states in future expansions.

## 7. User Flows
### 7.1 Main flow
Visitor lands on hero, understands the desktop companion, scans proof, checks features, verifies privacy, then opens GitHub/install.

### 7.2 Secondary flows
Security-minded visitors jump to Privacy. Developers jump to Architecture or Install.

### 7.3 Edge cases and error states
Avoid linking to missing assets or fictional demos. If a feature is future roadmap, label it as future rather than shipping.

## 8. Technical Recommendations
### 8.1 Suggested UI framework/library
Continue with Next.js, Tailwind, framer-motion, and lucide-react.

### 8.2 Responsive strategy
Hero should collapse to text first, then the runtime mock. Proof metrics must remain two columns on mobile.

### 8.3 Accessibility
Use descriptive image alt text, real links, strong contrast, and no text trapped inside tiny decorative badges.

### 8.4 Performance considerations
Use the existing optimized local assets, avoid missing image references, keep animations restrained, and prefer static sections for product proof.

## 9. Council Disagreements
Jony wanted to preserve more editorial warmth; Jobs wanted a harder developer page. The chosen direction keeps warmth in the palette and companion image, while putting the product proof in the foreground.

## 10. Next Steps
1. Replace outdated multi-page portfolio routes with real docs or redirect them to homepage anchors.
2. Add real screenshots or short videos of the Electron pet, settings, onboarding, and memory dashboard.
3. Add a security/privacy page once the project has a published data model and threat model.
4. Add release notes and changelog links from GitHub.

---
## Appendix: Debate Record
Norman pushed for cognitive clarity and truthful navigation. Ive defended emotional warmth but conceded trust must be visible. Jobs rejected vague creative-AI language and demanded that the first viewport explain the product immediately.
